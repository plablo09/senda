# Alembic code hygiene: modern type syntax, duplicate Makefile target, revision ID convention

## Problem Statement

Three small hygiene issues that don't affect correctness but create ongoing noise and set suboptimal patterns for future contributors.

## Findings

### 1. `alembic/script.py.mako` and `0001_initial_schema.py` use legacy `Union` type syntax

**Location:** `alembic/script.py.mako:10-11` and `alembic/versions/0001_initial_schema.py:10-11`

Both files use `from typing import Sequence, Union` and `Union[str, None]` syntax. The project is Python 3.12 — the correct syntax is `str | None`. The template file is especially important: every future migration generated by `make revision` will inherit the legacy pattern.

Fix in `script.py.mako`:
```python
# Remove:
from typing import Sequence, Union

# Update variable declarations:
down_revision: str | None = ${repr(down_revision)}
branch_labels: str | Sequence[str] | None = ${repr(branch_labels)}
depends_on: str | Sequence[str] | None = ${repr(depends_on)}
```

Note: `Sequence` can be imported from `collections.abc` if needed for runtime use, but in migration files it's annotation-only so the import can be dropped entirely.

Fix in `0001_initial_schema.py`: same change — remove `Union` import, update the three variable declarations.

### 2. `lint` and `fmt-check` Makefile targets are identical

**Location:** `Makefile:32-34` and `Makefile:40-42`

Both run exactly the same two commands:
```makefile
lint:
    docker compose run --rm api ruff check api/
    docker compose run --rm api black --check api/

fmt-check:
    docker compose run --rm api ruff check api/
    docker compose run --rm api black --check api/
```

Having two canonical names for the same operation creates confusion about which to call. One should be deleted, or one should delegate to the other.

Fix: Delete `fmt-check` entirely (or make it call `lint`). Update `.PHONY` declaration accordingly.

### 3. Non-standard revision ID `"0001"` will create friction with autogenerate

**Location:** `alembic/versions/0001_initial_schema.py:16`

Alembic generates revision IDs as 12-character hex hashes (e.g., `a3f8b2c19e04`). Using `"0001"` is human-readable but creates a decision point: either all future migrations must use sequential numeric IDs (requiring manual editing of every `--autogenerate` output), or the chain will mix ID formats. The `make revision` target generates standard hash IDs, which will have `down_revision = "some_hash"` pointing to `"0001"` — this works, but looks inconsistent.

Two paths forward:
- **Accept numeric IDs as project convention** — document in `AGENTS.md` that `down_revision` in autogenerated files must be manually updated if it points to `"0001"`, and that the project uses sequential numeric IDs.
- **Change to a hash ID** — rename `"0001"` to a proper 12-char hex string and update the filename accordingly.

Either is fine; what matters is that the decision is documented in `AGENTS.md`.

## Acceptance Criteria

- [ ] `script.py.mako` uses `str | None` syntax (no `Union` import)
- [ ] `0001_initial_schema.py` uses `str | None` syntax (no `Union` import)
- [ ] One of `lint` / `fmt-check` is removed from Makefile
- [ ] Revision ID convention documented in `AGENTS.md` (either numeric or hash, not both)
