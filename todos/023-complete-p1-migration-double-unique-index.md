# Fix Duplicate Unique Constraints in Migration 0002

## Problem Statement

Migration `0002` declares both a `UniqueConstraint` inside `create_table` AND a separate `create_index(..., unique=True)` on the same columns for both `usuarios.email` and `sesiones_refresh.jti`. PostgreSQL creates two distinct unique indexes per column ŌĆö redundant storage, double index maintenance on every write, and an unnamed constraint that is difficult to reference in future migrations. Since the DB is greenfield, the fix is to amend the migration directly.

## Findings

- `alembic/versions/0002_add_usuarios_and_refresh_sessions.py:45`: `sa.UniqueConstraint("email")` inside `create_table("usuarios")`
- `alembic/versions/0002_add_usuarios_and_refresh_sessions.py:47`: `op.create_index("ix_usuarios_email", "usuarios", ["email"], unique=True)` ŌĆö second unique index on same column
- `alembic/versions/0002_add_usuarios_and_refresh_sessions.py:63`: `sa.UniqueConstraint("jti")` inside `create_table("sesiones_refresh")`
- `alembic/versions/0002_add_usuarios_and_refresh_sessions.py:66`: `op.create_index("ix_sesiones_refresh_jti", ...)` ŌĆö second unique index
- Verification SQL: `SELECT indexname FROM pg_indexes WHERE tablename IN ('usuarios', 'sesiones_refresh')` will show duplicates on a deployed instance
- The `downgrade` already correctly references only the named indexes ŌĆö the anonymous `UniqueConstraint` creates an index that downgrade does not drop

## Proposed Solutions

### Option 1: Remove inline UniqueConstraints, keep named create_index (Recommended)

**Approach:** Delete the `sa.UniqueConstraint("email")` line from `create_table("usuarios")` and `sa.UniqueConstraint("jti")` from `create_table("sesiones_refresh")`. The named `create_index` calls already enforce uniqueness with stable, referenceable names.

**Pros:**
- Single unique index per column (correct)
- Named index gives stable name for future migrations and error messages
- Downgrade already correct as-is
- Since DB is greenfield, amend migration file directly ŌĆö no 0003 needed

**Cons:**
- Must re-run migrations on any existing dev environment (wipe DB or manually drop the orphaned anonymous constraints)

**Effort:** 15 minutes

**Risk:** Low (greenfield DB)

## Recommended Action

Amend `0002` directly. Wipe and re-run migrations on dev (`make migrate-down && make migrate`). The anonymous constraint names are auto-generated by PostgreSQL and not tracked by Alembic, so a corrective migration would be harder than just fixing the source.
