# LLM Student Guidance (Phase 3)

## Problem Frame

Students working through geographic/statistical programming exercises need guidance when they get stuck — enough to lower the cognitive burden of syntax and debugging so they can focus on learning the analytical concepts. Too much LLM help hampers understanding; too little builds frustration. The system must strike a pedagogically sound balance.

## Requirements

- R1. When a student's code execution produces an error, the system triggers an LLM feedback request on the **first** error for a given exercise.
- R2. After delivering feedback, the system enters a **silence window**: no LLM feedback is triggered for the next N student attempts on the same exercise (N is configurable, default: 2).
- R3. After the silence window, the next error triggers another feedback response. This cycle repeats until a **hard limit** of total feedback responses per exercise is reached (configurable, default: 3).
- R4. Once the hard limit is reached, no further LLM feedback is triggered for that exercise in the current session.
- R5. Attempt and feedback counts are tracked server-side in Redis, keyed to `(exercise_id, session_token)`. Session token is a UUID generated by the browser on first visit and persisted in localStorage.
- R6. LLM requests are routed via LiteLLM using Gemini (default: `gemini/gemini-2.0-flash`) via the student's API key configured in the environment. Provider is fully swappable via `LLM_MODEL` / `LLM_API_BASE` / `LLM_API_KEY` env vars.
- R7. Ollama remains in `docker-compose.yml` but is disabled by default (commented out or behind a compose profile) to reduce local dev startup complexity.
- R8. The API key is never included in rendered student HTML — validated by an integration test.
- R9. Every code execution error is logged to a simple DB table (`execution_errors`: `documento_id`, `ejercicio_id`, `session_token`, `error_type`, `timestamp`). No teacher UI in this phase.
- R10. All LLM-generated feedback text is in Spanish.

## Success Criteria

- A student who errors on attempt 1 sees a hint; attempts 2–3 produce no hint; attempt 4 error triggers a second hint.
- A student who hits the hard limit sees a clear message (in Spanish) that no more hints are available for that exercise.
- Switching from Gemini to Ollama requires only an env var change — no code changes.
- No API key appears in any rendered HTML (asserted in CI).
- Error events are queryable in the DB with no additional instrumentation needed when teacher analytics UI is built later.

## Scope Boundaries

- No teacher-facing analytics UI in this phase. Data collection only.
- No student authentication. Session identity is a browser-generated UUID.
- No time-based rate limiting — attempt count only.
- No LLM assistance on the teacher authoring side.
- Ollama integration is deferred; it stays in compose but is not the active default.

## Key Decisions

- **Gemini over Ollama as active default:** Reduces container orchestration complexity and unblocks user-facing testing earlier. Ollama remains available via env var.
- **Graduated intervention over flat rate limit:** Pedagogically motivated — forces independent attempts between hints rather than allowing rapid hint-farming.
- **Attempt-count over time-based silence:** Students work on their own schedule; time-based windows would penalize careful, interrupted work.
- **Log-only analytics:** Captures the data now at near-zero cost; teacher heatmap UI deferred until teacher-facing work resumes.
- **Soft session identity (UUID):** No auth in Phase 3. Limits are soft — a page reload resets the session — which is acceptable for the current stage.

## Dependencies / Assumptions

- Redis is already running and used for Celery + render pub/sub. Feedback state uses the same Redis instance.
- LiteLLM is already declared as a dependency (`litellm>=1.50` in `pyproject.toml`).
- Gemini free tier is sufficient for the expected student load in early testing.
- `ejercicio_id` is a stable, addressable identifier on the exercise block — needs to be confirmed during planning.
